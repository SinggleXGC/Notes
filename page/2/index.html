GET请求和POST请求的区别？

发表于 2021-06-10 分类于 面试题 ， 计算机网络面试题
  1. GET把参数包含在URL中，POST把参数放在请求体中。这样，POST能够发送的数据更大。(URL有长度限制)
  2. GET请求比POST请求快
    • POST请求包含更多的请求头(但可忽略不计)
    • 最重要的是：POST在真正发送数据的时候，会先将请求头发送给服务器进行确认，然后才发送数据
  3. GET将数据缓存起来，而POST不会
  4. POST请求一般用于修改和写入操作，而GET请求用来获取资源操作。
  5. POST请求更加安全(参数不放在URL、数据不缓存)
  6. POST能够发送更多的数据类型(GET只能发送ASCII字符)

POST请求的过程：

  1. 在TCP的第三次握手，浏览器会进行确认并发送POST请求头。
  2. 服务器返回100 Continue响应
  3. 浏览器发送数据
  4. 服务器返回200 OK响应

GET请求的过程：

  1. 在TCP第三次握手，浏览器就会进行确认并发送GET请求头和数据
  2. 服务器返回200 OK响应

这样GET的总耗是POST的2/3左右。

TCP三次握手和四次挥手

发表于 2021-06-09 更新于 2021-06-10 分类于 计算机网络

TCP报文格式

在讲TCP三次握手和四次握手之前，我们需要先了解TCP报文的格式。

  • Sequence Number：序列号。当SYN=1时，该字段表示该报文段的第一个字节的编号。当SYN不为1时，该字段表示初始序列号，用于同步序列号。
  • Acknowledgment Number：确认号。表示接受方期望收到发送方下一个数据包的第一个字节的编码。
  • SYN：在建立连接时使用，用于同步序列号。
  • ACK：表示确认号是否有效。
  • FIN：表示数据是否发送完毕。

TCP三次握手

TCP建立连接的过程叫做握手，握手在接收方和发送方之间需要发送三次报文段。

  1. 一开始，客户端处于closed状态，服务端处于listen状态。
  2. 客户端从服务端发送一个TCP报文，SYN=1，seq=x，用于向服务端请求建立TCP报文。此时，客户端处于SYN-SENT状态。
  3. 服务端接收到客户端的连接请求报文之后，向客户端发送一个确定报文。ACK=1，ack=x+1，SYN=1，seq=y。此时服务端处于SYN-RCVD状态。
  4. 客户端在接收到服务端发送的确认报文之后，再向服务端发送一个确认报文。ACK=1，seq=x+1，ack=y+1。此时客户端处于ESTABLISHED状态。
  5. 服务端接收到客户端的确认报文之后，进入ESTABLISHED状态。到这里，TCP连接就建立成功，可以进行数据传送。

TCP为什么还需要第三次握手？

第三次握手就是客户端对服务端的确认进行确认。

那么为什么客户端最后还要进行一次确认呢？

这是为了避免已失效的连接请求报文段突然又传送到了服务端，因而产生操作。

现考虑这么一种情况：

客户端发送了一次连接请求报文，但因报文丢失导致长时间没有收到服务端的确认。因此，客户端重新发送了一次连接请求报文，后来收到了确认，建立了连接。数据传输完毕之后，就释放了连接。

假如，客户端第一次发送的连接请求报文并没有丢失，而是因为网络阻塞导致延迟到了连接释放之后才到达。那么，服务端接收到延迟的连接请求报文之后，误以为客户端发送了一次连接请求报文。就会向客户端发送确认报文。

假如不采用第三次握手，那么在服务端发送确认报文的时候，连接就建立了。但是，客户端因为没有发送连接请求报文，因此会忽略服务端的确认。而服务端却以为连接建立了，一直在等待客户端发送数据。这样服务端的许多资源就浪费了。

TCP四次挥手

  1. 一开始，客户端和服务端都处于连接建立成功的状态。
  2. 客户端向服务端发送连接释放报文。FIN=1，seq=u。这时客户端进入FIN-WAIT-1状态(终止等待1状态)。
  3. 服务端接收到客户端发送的连接释放报文，会对客户端发送确认。ACK=1，seq=v，ack=u+1。服务器进入到CLOSE-WAIT状态。此时，TCP服务器进程通知高层应用程序，从客户端到服务端方向的连接释放了，此时的TCP连接处于半关闭状态(half-close)状态。
  4. 客户端接收到服务器发送的确认之后，进入到FIN-WAIT-2状态，等待服务器发送的数据。
  5. 若服务器已经发送完数据了，就向客户端发送报文。FIN=1，seq=w，ACK=1，ack=u+1。此时，服务端进入LAST-ACK状态。
  6. 客户端接收到服务器发送的报文后，向服务器发送确认。ACK=1，seq=u+1，ack=w+1。接着客户端进入TIME-WAIT状态，等待2MSL(Maximum Segment Lifetime，最长报文段寿命)。在经历2MSL之后，客户端进入CLOSED状态。
  7. 服务器收到确认之后，就会进入CLOSED状态。
  8. 这样TCP的连接就释放了。
